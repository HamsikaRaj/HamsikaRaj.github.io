Agentic AI

Research-and-Report Agents

A three-agent research system on the OpenAI Agents SDK with typed handoffs, guardrails, and an LLM-judge eval gate.

  • OpenAI Agents SDK
  • MCP
  • FastAPI
  • Guardrails
  • Pytest
  • SSE
View code

Problem

Research assistants that free-text their way to an answer are hard to trust and impossible to regression-test. I wanted typed contracts, grounded evidence, and an automated quality gate, so every change is provably not a regression.

My role

Sole designer and engineer, covering agent architecture, the MCP server, guardrails, and the eval harness.

Process

Built a three-agent pipeline on the OpenAI Agents SDK: a Planner delegates via typed handoffs, an Executor gathers grounded evidence through function tools and a custom MCP server, and a Reviewer returns a guardrail-checked, typed ReportResult. Added input/output guardrails, a custom tracing processor, and a golden-set eval harness behind a pytest gate.

Solution & impact

  • Typed handoffs and a typed ReportResult make the pipeline's contracts explicit and testable end-to-end.
  • Input/output guardrails enforce PII redaction and groundedness before any output ships.
  • A golden-set LLM-judge eval runs behind a pytest regression gate, so quality drops fail CI instead of reaching users.
  • Streams token-by-token over both a CLI and an SSE endpoint, with a custom tracing processor for tokens and latency.

What I learned

Typed contracts plus an automated eval gate turn an agent from a demo into something you can safely change. The guardrails and regression gate are what make fast iteration possible without breaking trust.

Agentic AI

Multi-Agent Coding Assistant

Five specialized Claude agents that turn one prompt into a reviewed, multi-file codebase.

  • Python
  • LangGraph
  • FastAPI
  • Anthropic API
  • React
  • TypeScript
  • SSE
View code

Problem

Turning a plain-English request into working software usually means one model doing everything, and quietly shipping broken code. I wanted a pipeline that plans, builds, and reviews itself before delivering.

My role

Sole designer and engineer, covering architecture, agent orchestration, cost strategy, and the full-stack UI.

Process

Built a 5-agent LangGraph pipeline (Planner → Architect → Coder → two independent Reviewers) with a self-correcting loop that returns failing code to the coder with specific feedback (up to 3 retries). Routed models by task to control cost and streamed per-agent telemetry over SSE.

Solution & impact

  • Shipped a 25-file React/TypeScript app in a single run, with the review loop catching and repairing broken output before it shipped.
  • Cut inference cost via per-task routing (Opus for generation ≈82% of cost, Sonnet for planning/review, where a full dual-reviewer cycle ran $0.0007), fronted by a Haiku intent classifier.
  • Telemetry surfaced the real bottleneck: ~8.7s coder vs ~1.5s reviewer.

What I learned

Reliability in agentic systems comes from structure and feedback loops, not bigger models. Routing the right model to the right task beats defaulting to the most expensive one everywhere.

RAG Evaluation Harness

Automated, claim-level evaluation of RAG pipelines using Claude as an LLM-judge.

  • Python
  • FastAPI
  • FAISS
  • Claude API
  • Sentence Transformers
  • Pydantic
  • Pytest
View code

Problem

Retrieval metrics tell you the right documents came back, not whether the answer was faithful or hallucinated. Manual scoring doesn't scale. RAG teams need automated, trustworthy evaluation.

My role

Designer and engineer of the full harness: retrieval metrics, LLM-judge scoring, and the multi-backend gateway.

Process

Used Claude as an LLM-judge to score faithfulness, answer relevancy, and hallucination at the claim level, and built a pluggable retrieval layer (FAISS, ChromaDB, Qdrant, Weaviate) behind a FastAPI gateway so cross-backend benchmarking is a one-config change.

Solution & impact

  • Cut evaluation effort ~90% vs. manual scoring.
  • On a 75-question benchmark: faithfulness 0.996, answer relevancy 0.82, hallucination 0.017 (98 claims verified, 0.98 grounding).
  • Retrieval: Recall@5 0.87 / Recall@10 0.92, MRR 0.75, NDCG@10 0.79.

What I learned

You can't improve what you can't measure. Building the eval harness first changes how you build the system. It turns "seems fine" into a number you can defend.

RAG · Security

FinSolve: RAG with Role-Based Access Control

An internal RAG chatbot where access control is enforced at the vector-search level.

  • Python
  • FastAPI
  • Qdrant
  • Claude Sonnet
  • Sentence Transformers
  • JWT
  • Streamlit
View code

Problem

A finance employee should be able to ask about quarterly revenue, but should never see HR records. Prompt-level guardrails leak. Access control had to happen at retrieval, not in the prompt.

My role

End-to-end designer: auth pipeline, ingestion, retrieval tuning, and adversarial testing.

Process

Built a JWT auth pipeline deriving roles dynamically from a 100-person HR dataset, filtered retrieval by role/department in Qdrant, and pressure-tested it with adversarial queries.

Solution & impact

  • Dual-layer RBAC across 6 role tiers and 5 department knowledge bases, reaching 100% access-control accuracy, 0 hallucinations across 16 adversarial tests.
  • Ingestion chunked 9 sources (~1,700 lines) into 225 metadata-tagged vectors, with retrieval tuned to sub-second RBAC-filtered results.
  • Replaced a 1.2B local Llama 3.2 model that hallucinated on 100% of out-of-scope queries with Claude Sonnet, driving fabricated responses to zero.

What I learned

Security in AI systems is an architecture decision. Enforcing it at the data layer is the difference between a demo and something an enterprise can actually deploy.
